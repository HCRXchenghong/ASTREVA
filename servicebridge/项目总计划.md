# 项目总计划

本文档记录在线客服系统的新执行计划。当前产品方向是：访客使用网站 Web 聊天窗口，人工客服直接在飞书中接待和回复，不再建设独立客服 App / 管理 App。

## 1. 总目标

做一个轻量、高效、纯自研的在线客服系统。

端形态：

- 用户端：Web 页面，支持电脑浏览器和手机浏览器。
- 人工客服端：飞书群聊 / 飞书机器人 / 飞书事件回调。
- 运维管理：后端 API、配置文件、审计与导出接口；不再维护独立移动管理端。

性能目标：

- 不考虑带宽瓶颈时，系统按 5000 并发 WebSocket 连接容灾设计。
- WebSocket 节点可横向扩容。
- 消息先落库再投递。
- 单节点异常时不丢已确认消息。
- 飞书回调必须幂等，避免事件重试造成重复客服消息。

## 3. 后端模块

核心模块：

- 访客模块：访客 ID、访客 IP、默认备注。
- 会话模块：会话创建、状态流转、当前处理方。
- 消息模块：文本、表情、联系方式、系统消息、AI 消息。
- WebSocket 网关：访客鉴权、心跳、重连、消息投递。
- 飞书直连模块：获取 `tenant_access_token`、发送访客消息到飞书、接收飞书消息事件、会话映射、幂等落库。
- AI 模块：OpenAI 原生 API 和 OpenAI 兼容协议适配。
- 关键词模块：关键词自动回复、人工客服关键词。
- 营业时间模块：人工优先、非营业时间 AI、超时转 AI。
- 管理模块：配置、会话监控、聊天记录、审计与 CSV 导出。

## 4. 必须实现

用户端 Web：

- 手机浏览器全屏聊天体验。
- 电脑浏览器居中聊天体验。
- 文本、表情、更多面板。
- 电话、微信、QQ / WhatsApp 联系方式触发。
- AI / 人工 / 排队 / 已结束状态。
- 转人工。
- 历史消息加载入口。
- 服务评价。

飞书人工客服：

- 访客新消息推送到默认飞书群。
- 飞书消息保留 `#conv_xxx` 会话号。
- 客服直接回复飞书消息，后端自动映射到原会话。
- 不在回复线程中时，客服可在消息中带 `#conv_xxx` 指定会话。
- 飞书回复落库为客服消息，并实时推送给网站访客。
- 飞书回调按 `event_id/message_id` 幂等处理。
- 后端维护虚拟客服 `agent_feishu`，用于归属会话、审计和统计。

飞书配置通过后台接口保存到数据库：

- `GET /api/admin/integrations/feishu`
- `PATCH /api/admin/integrations/feishu`
- `POST /api/admin/integrations/feishu/test`

环境变量 `FEISHU_*` 仅作为首次启动兜底，不作为最终配置入口。

## 5. 开发阶段

### 阶段一：飞书直连替换

目标：

- 删除独立客服 App / 管理 App 目录。
- 建立飞书 API 客户端。
- 建立飞书事件回调接口。
- 建立飞书消息和本地会话的绑定表。

完成标准：

- 后端可启动。
- 飞书 URL 校验可返回 challenge。
- 访客消息可发送到飞书群。
- 飞书回复可回到网站访客。

### 阶段二：飞书运营体验

目标：

- 飞书消息格式更清晰。
- 支持飞书卡片按钮：接管、关闭、查看摘要。
- 支持失败重试和告警。

完成标准：

- 客服在飞书里可以完成主要接待动作。
- 飞书回调失败不会丢失访客消息。

### 阶段三：容量和上线

目标：

- 多 WebSocket 节点。
- Redis 在线状态和跨节点投递。
- 压测脚本。
- Nginx WebSocket 和飞书回调反代配置。

完成标准：

- 5000 并发连接压测不崩溃。
- 心跳稳定。
- 消息投递稳定。
- 单节点异常后可重连恢复。

## 6. 不变原则

- 访客侧体验不能退化。
- 消息必须先落库再投递。
- 飞书回调必须幂等。
- 人工客服关键词优先级最高。
- 营业时间内默认人工优先。
- 生产敏感配置不得明文暴露到响应中。
- 5000 并发按横向扩容和容灾设计，不按单机硬扛设计。
