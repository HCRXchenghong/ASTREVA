# 客服系统边界说明

## 1. 产品边界

本系统用于网站访客在线咨询。访客只接触网站 Web 聊天窗口；人工客服只在飞书里接待和回复。不再建设独立客服 App、独立管理 App，也不再以 App 推送作为主通知链路。

## 2. 系统内职责

系统负责：

- 创建访客和会话。
- 记录访客 IP，并作为默认备注。
- 保存访客消息、客服消息、AI 消息和系统消息。
- 通过 WebSocket 实时推送访客侧消息。
- 将访客消息推送到飞书群。
- 接收飞书消息事件，并把飞书回复转成客服消息。
- 维护飞书消息 ID 与本地会话 ID 的映射。
- 提供 AI 接待、关键词回复、营业时间和超时转 AI 能力。
- 提供会话、审计、评价、导出等后端管理接口。
- 支持 5000 WebSocket 并发的横向扩容和容灾设计。

系统不负责：

- 独立客服移动 App。
- 独立管理移动 App。
- 飞书之外的多渠道聚合。
- 飞书应用创建、企业授权审批、权限开通。
- 飞书历史消息的全量归档，只归档进入本系统会话链路的消息。

## 3. 飞书接待边界

飞书作为人工客服工作台。

必须支持：

- 访客新消息发送到默认飞书群。
- 飞书消息中带本地会话号 `#conv_xxx`。
- 客服回复飞书消息时，后端根据 `parent_id/root_id/message_id` 映射本地会话。
- 客服不在回复线程中时，可以手动带 `#conv_xxx` 指定会话。
- 飞书回复落库为 `agent` 消息，并实时推送给网站访客。
- 飞书消息事件按 `event_id` 或 `message_id` 幂等处理。
- 飞书 URL 校验使用 `verification_token`。
- 飞书事件加密使用 `FEISHU_ENCRYPT_KEY` 解密。
- 飞书应用凭 `app_id/app_secret` 获取 `tenant_access_token`。

当前不做：

- 飞书通讯录客服个人身份映射。第一阶段统一归属到虚拟客服 `agent_feishu`。
- 飞书卡片按钮强交互。后续可加接管、关闭、摘要按钮。

## 4. 访客端边界

用户端是 Web 页面。

必须支持：

- 手机浏览器全屏聊天。
- 电脑浏览器居中聊天。
- 文本消息。
- 表情消息。
- 常用联系方式快捷入口。
- AI / 人工 / 排队 / 已结束状态。
- 服务评价。
- 历史消息加载。

访客端不知道客服是否使用飞书，也不依赖飞书。

## 5. 后端边界

后端是系统核心事实源。

必须满足：

- 消息先落库再推送。
- 访客 WebSocket 断线后可补拉历史。
- 飞书回调不可直接信任，必须校验 token。
- 飞书回调失败时返回非 2xx，让飞书重试。
- 重复飞书事件不会生成重复客服消息。
- PostgreSQL 为生产事实库。
- Redis 只负责跨节点实时投递和状态同步，不作为消息事实库。

## 6. 配置边界

飞书生产配置通过后台保存：

- `GET /api/admin/integrations/feishu`
- `PATCH /api/admin/integrations/feishu`
- `POST /api/admin/integrations/feishu/test`

数据库保存位置为 `feishu_settings`。密钥字段不明文返回；配置 `DATA_ENCRYPTION_KEY` 后加密落库。`FEISHU_*` 环境变量只作为首次启动兜底。

飞书开放平台侧需要：

- 自建应用。
- 开启机器人能力。
- 给应用开通读取/接收消息事件和发送消息能力。
- 将事件订阅回调地址配置为 `/api/integrations/feishu/events`。
- 将应用添加到默认客服群。

## 7. 验收边界

上线前必须验收：

- 飞书 URL 校验通过。
- 访客发送消息后，飞书群收到消息。
- 客服在飞书回复后，网站访客实时收到回复。
- 同一飞书事件重复投递不会重复落库。
- 后端重启后，已绑定的飞书消息仍可映射回会话。
- 关闭飞书配置时，访客侧聊天不白屏。
- 5000 WebSocket 压测通过。
